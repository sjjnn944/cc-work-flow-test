<Purpose>
AI 협업 개발 과정에서 축적되는 교훈을 체계적으로 관리하는 스킬이다.
교훈을 `doc/base/ai-dev-guidelines/` 디렉토리에 파일로 저장하고, 관련 가이드 문서에의 반영을 검토하여 사용자 승인 후 적용한다.

세 가지 설계 제약:
1. **게으름 친화** — 교훈을 빠르게 쌓아두고 나중에 정리 가능
2. **교차 참조** — 하나의 교훈이 여러 가이드에서 필요할 수 있음
3. **단일 원천** — 교훈 원본은 ai-dev-guidelines/ 에만 존재, 가이드에는 참조 링크
</Purpose>

<Do_Not_Use_When>
- 기존 가이드 문서의 오타나 서식만 수정할 때
- 프로젝트별 CLAUDE.md 규칙을 추가할 때 (프로젝트 범위, doc/base 범위가 아님)
- 기존 교훈 파일의 내용만 수정할 때 (직접 편집)
</Do_Not_Use_When>

<Important>
`doc/base`는 git submodule (`project-init-guide.git`)이다. 모든 파일 변경은 submodule 내에서 이루어져야 한다.
</Important>

## 실행 워크플로우

### Step 1: 교훈 내용 파악

사용자 입력에서 다음을 추출한다:

| 항목 | 설명 | 예시 |
|------|------|------|
| 교훈 제목 | 한 줄 요약 | "소비자 모듈은 프로토콜 정의를 복사하지 않는다" |
| 배경 | 왜 이 교훈이 필요한지 | "TRAY implementation.md가 CORE의 IPC 정의를 인라인 복사..." |
| 적용 대상 | 어떤 단계/상황에 적용되는지 | "상세 설계 시, 구현 시" |

사용자가 충분한 정보를 제공하지 않으면 질문하여 보충한다.

### Step 2: 교훈 파일 생성

`doc/base/ai-dev-guidelines/{slug}.md` 파일을 생성한다.

**slug 생성 규칙**:
- 영문 소문자 + 하이픈
- 핵심 키워드 2~4단어
- 예: `no-protocol-copy`, `design-input-validation`

**파일 형식**:
```markdown
## 원칙

(교훈의 핵심 규칙을 명확하게 기술)

## 배경

(이 교훈이 도출된 맥락, 발생한 문제, 근본 원인)

## 적용 대상

(이 교훈이 적용되는 워크플로우 단계, 가이드 문서, 상황)

## 관련 가이드

(반영 대상 가이드 목록 — Step 3에서 확정 후 업데이트)
```

### Step 3: README.md "미적용" 테이블에 등록

`doc/base/ai-dev-guidelines/README.md`의 "미적용" 테이블에 행을 추가한다:

```markdown
| {교훈 제목} | [{slug}.md]({slug}.md) | {적용 대상} | {비고} |
```

### Step 4: 가이드 반영 분석

다음 가이드 문서를 읽고, 교훈이 반영되어야 하는 위치를 분석한다:

| 가이드 | 경로 | 주요 영역 |
|--------|------|----------|
| 상세 설계 가이드 | `doc/base/detailed-design-guide.md` | 설계 원칙, 문서 작성 규칙 |
| 폴더 구조 가이드 | `doc/base/folder-structure-guide.md` | 디렉토리 구조, 파일 배치 |
| 구현 가이드 | `doc/base/implementation-guide.md` | 코딩 패턴, 에러 처리 |
| 시스템 설계 가이드 | `doc/base/system-design-guide.md` | 아키텍처, 모듈 분리 |
| 테스트 설계 가이드 | `doc/base/test-design-guide.md` | TC 작성, 검증 방법 |
| 검증 가이드 | `doc/base/verification-guide.md` | 검증 단계, 품질 기준 |

**분석 결과를 사용자에게 보고한다**:

```
## 반영 분석 결과

### 반영 대상
- `implementation-guide.md` 2.3절: (구체적 변경 내용)
- `detailed-design-guide.md` 4절: (구체적 변경 내용)

### 반영 불필요
- `folder-structure-guide.md`: (사유)
- ...
```

**모든 가이드를 읽어야 한다** — 교훈이 어디에 영향을 미칠지 사전에 단정하지 않는다. 예상 밖의 가이드에 반영이 필요할 수 있다.

### Step 5: 사용자 승인

반영 분석 결과를 보여주고 사용자 승인을 받는다.

사용자가 선택할 수 있는 옵션:
- **전체 승인** — 모든 제안 반영
- **부분 승인** — 특정 가이드만 선택
- **등록만** — 가이드 반영 없이 "미적용"으로 남김 (나중에 반영)
- **수정 요청** — 변경 내용 조정 후 재승인

### Step 6: 가이드 반영 실행

승인된 가이드에 교훈을 반영한다.

**반영 원칙**:
- 가이드 본문에 규칙을 직접 작성한다 (교훈 파일 참조 링크가 아닌 실제 내용)
- 기존 문서의 구조와 서술 스타일에 맞춘다
- 관련 절에 자연스럽게 통합한다 (별도 섹션 추가보다 기존 절에 삽입 우선)

### Step 7: README.md 테이블 이동

가이드 반영이 완료된 경우:
1. "미적용" 테이블에서 해당 행 제거
2. "적용됨" 테이블의 적절한 Part 섹션에 행 추가
3. 적용 가이드 컬럼에 반영된 가이드 약칭 기재

가이드 반영 없이 "등록만" 한 경우:
- "미적용" 테이블에 유지
- 비고 컬럼에 상태 기재

### Step 8: 변경 요약 보고

완료 후 사용자에게 보고한다:

```
## 완료 요약

### 생성된 파일
- `doc/base/ai-dev-guidelines/{slug}.md`

### 수정된 파일
- `doc/base/ai-dev-guidelines/README.md` — 인덱스 등록
- `doc/base/{guide}.md` — {변경 요약}
- ...

### submodule 커밋 안내
변경사항은 `doc/base` submodule 내에 있습니다.
커밋이 필요하면 말씀해 주세요.
```
